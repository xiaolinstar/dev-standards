## 背景

随着 `drink-budget`、`party-helper`、`ai-todo` 等项目多端（C 端前台、营销落地页、B 端运营后台、小程序内嵌
H5）迭代，高频移动端（Mobile H5）访问与兼容性问题开始凸显：

- **缺乏跨项目规范**：各个 H5 页面在视口适配、首屏加载、组件手感、微信分享、软键盘遮挡等方面标准不一。
- **环境差异复杂**：iOS 与 Android 在不同浏览器（WeChat WebView、钉钉、原生 Safari/Chrome）中的行为差异（
  例如 iOS 软键盘收起后高度塌陷、弹性橡皮筋回弹等），导致了大量的兼容性 Bug。
- **UI 风格与手感分裂**：输入框触控面积过小、下拉菜单在不同机型展示迥异、分页按钮在触屏下极难操作。

## 决策

为 H5 应用定义通用 Playbook 文件 [playbook/h5.md](../h5.md) 作为所有 H5 项目的唯一真相基线。核心选型与规约包括：

### 1. 视口适配以 vw 为核心，向下兼容 PC 沙盒

**理由**：

- 摒弃 Rem/Lib-flexible 方案。采用原生的 Viewport 单位（vw/vh）在编译时（PostCSS 插件）自动进行适配。
- 设定设计稿宽度基准为 `375`（iPhone SE/8/X 宽度）。
- 为兼容 PC 端预览与测试，采用居中沙盒模式（限宽 375px~480px 并配以 shadow 和留白）。

### 2. 强力约束移动端组件（严禁原生下拉选择）

**理由**：

- **防分裂**：不同系统下原生 `select`、`input[type="date"]` 等控件展现差异极其悬殊，极易造成布局崩溃。
  因此强制一律使用“只读 Field 触发器 + 底部弹出 Popup + Picker 选择器”的移动端交互组合。
- **触控面积**：为保障高容错率，所有按钮和点击区域的最小触控高度不得低于 `44px`。
- **上拉无限滚动**：明令禁止在触屏上使用 PC 端的分页按钮，统一采用下拉刷新与上拉无限加载列表。

### 3. 首屏性能优化作为强制指标

**理由**：

- 移动端网络环境复杂（3G/4G/5G/Wi-Fi 切换），用户对页面加载速度极其敏感。
- 强制指标：非首屏资源必须通过路由懒加载（Code Splitting）进行按需加载。
- 资源处理：图片默认采用 WebP 格式，非首屏图片使用 `loading="lazy"`。对于 LCP（最大内容渲染）英雄图，
  必须设置 `fetchpriority="high"` 提高加载优先级。

### 4. 彻底解决移动容器与软键盘兼容性历史宿疾

**理由**：

- **软键盘塌陷**：在 iOS 微信内，输入法弹起再收起后，页面底部往往留白且点击位置发生物理偏移。
  规范要求在输入框 `blur` 时强制触发一次页面微滚动（如 `window.scrollTo(0, 0)`）来修正点击偏移。
- **弹性回弹控制**：弹窗与浮层激活时，必须静止底层 Body 的滚动，防止出现双重橡皮筋滚动冲突。

## 范围

**覆盖**：

- 视口尺寸适配（Viewport、vw 转换、PC 沙盒）
- 核心 UI 交互标准（触控尺寸、表单输入、下拉选择器规范、滚动分页）
- 移动端首屏性能优化（LCP 优化、资源按需加载、图片格式）
- 移动浏览器容器兼容（微信 JS-SDK、软键盘避让、橡皮筋防抖）

**不覆盖**：

- 后台专有业务逻辑（如 RBAC 动态路由过滤）→ 见子文档 [playbook/h5-admin.md](../h5-admin.md)

## 后果

**正面**：

- 未来 `drink-budget`、`party-helper`、`ai-todo` 等项目可以共享同一套 H5 底座与设计规范。
- 极大地降低了移动端各种软键盘、微信 WebView 授权、滚动崩溃的调试耗时。

**负面 / 风险**：

- 现有已上线的 H5 页面如果不符合该规范，可能面临重构补差。
